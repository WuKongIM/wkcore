# 分布式后台管理系统 UI 设计

- 日期：2026-04-08
- 状态：已确认的第一版设计草案
- 面向项目：`WuKongIM`
- 设计目标：为 `单集群` 部署形态提供一套面向 `SRE / 运维值守` 的后台管理 UI，优先解决巡检和值班场景下的快速发现问题能力。

## 1. 背景与上下文

当前仓库已经具备分布式运行时的核心结构，包括：

- `pkg/cluster/raftcluster`：集群路由、转发与发现
- `pkg/replication/multiraft` / `pkg/replication/isr` / `pkg/replication/isrnode`：复制与一致性运行时
- `internal/runtime/online`：在线会话注册与本地投递
- `internal/access/gateway`：接入层协议适配
- `internal/access/api`：极简 HTTP API 入口

从现状看，项目已经有清晰的节点、复制组、网关、消息投递等概念，但现成的管理 API 仍很薄，当前 HTTP 入口主要集中在：

- `/healthz`
- `/user/token`
- `/message/send`

这意味着第一版后台设计不能假设已经存在完备的管理控制面。UI 需要围绕未来的观测聚合接口设计，而不是绑定当前零散的内部结构体。

另一个重要约束是：本项目没有独立的“单机模式”，单节点部署统一视为“单节点集群”。因此后台在文案、部署视角和信息架构上统一采用“集群”语义。

## 2. 设计结论

本轮讨论确认的产品边界如下：

- 主要用户：`运维 / SRE`
- 管理范围：`单集群`
- 第一版定位：`观测优先`
- 关注重点：`集群内核 + 接入流量`
- UI 形态：`高信息密度的日常运维工作台`
- 核心诉求：`巡检值守`
- 时间视角：`实时状态 + 短趋势（1h / 24h）`

产品方向上，对比过三类方案：

1. 纯巡检驾驶舱
2. 巡检 + 深度排障台
3. 巡检 + 轻量运维操作台

最终选择：

- 第一版采用 `纯巡检驾驶舱`
- 但在信息架构和详情页结构上预留未来向深度排障台扩展的路径

这样做的原因是：

- 最符合当前阶段的真实需求：先让 SRE 快速知道集群是否健康
- 不会把 UI 设计绑死在尚未建设完成的控制面 API 上
- 能把第一版范围压缩到真正高价值的可观测能力

## 3. 产品目标

第一版后台的目标不是“什么都能管”，而是让 SRE 在值守场景下完成下面三件事：

1. 在 30 秒内判断单集群是否整体健康
2. 快速发现异常节点、异常复制组、异常接入面
3. 通过 1h / 24h 短趋势判断问题是瞬时抖动还是持续恶化

第一版不追求：

- 完整排障平台
- 运维变更控制台
- 用户运营后台
- 长周期容量规划中心

## 4. 信息架构

第一版后台建议采用 7 个一级模块：

### 4.1 Dashboard

单集群总览驾驶舱，用于完成日常巡检的第一眼判断。

### 4.2 Nodes

按节点查看健康状态、承载角色、接入压力和资源占用。

### 4.3 Groups

按复制组查看 leader、副本健康、复制滞后与稳定性。

### 4.4 Gateway

查看 listener、连接接入、在线会话以及网关异常。

### 4.5 Traffic

查看消息流量、投递成功率、ack、热点与失败走势。

### 4.6 Events

聚合节点、复制组、接入层和流量面的异常事件与告警。

### 4.7 Topology

只读查看当前集群配置、节点分布、group 分布和监听配置。

## 5. 各模块详细设计

## 5.1 Dashboard

首页采用“三段式高密度工作台”，回答三个问题：

- 现在整体健康吗
- 异常集中在哪
- 趋势是在变好还是变坏

### 顶部状态带

包含：

- 集群名称
- 当前时间
- 自动刷新开关
- 时间窗切换：`实时 / 1h / 24h`
- 全局搜索入口

### 核心状态卡

建议放置如下卡片：

- 集群健康分
- 存活节点数
- 异常 Groups 数
- Leader 偏斜度
- Listener 异常数
- 当前连接数
- 在线会话数
- 消息入吞吐
- 消息出吞吐
- 失败率

要求：

- 每张卡片可点击
- 点击后可联动筛选下方表格或跳转对应模块

### 中部风险区

左侧放 `当前风险清单`，显示 5 到 8 条最重要的风险项，例如：

- 节点离线
- leader 频繁切换
- 复制 lag 超阈值
- listener 异常
- 连接突降
- 消息失败率突升

右侧放 `集群结构快照`，建议使用信息密度较高的状态矩阵，而不是装饰性拓扑图。适合的表达是：

- `节点 x 组` 状态矩阵
- leader 分布热度
- 异常 group 聚集节点

### 下部趋势与排行

趋势图建议只保留最核心的 4 组：

- 连接数
- 在线会话数
- 消息吞吐
- 复制 lag / 失败

同时放置两张高密度表：

- `异常节点排行`
- `异常 Groups 排行`

两张表都需要支持：

- 排序
- 关键列筛选
- 直接下钻到详情抽屉或详情页

## 5.2 Nodes

`Nodes` 页是按机器或实例巡检的核心页面。

### 页面结构

- 顶部汇总区
- 中部高密度节点表
- 右侧详情抽屉

### 顶部汇总指标

- 节点总数
- 在线数
- 疑似异常节点数
- 平均负载
- Leader 最重节点
- 连接数最高节点

### 主表格字段建议

- `Node ID`
- `节点名`
- `状态`
- `Leader / Follower 摘要`
- `Groups 数`
- `连接数`
- `在线会话数`
- `消息入速率`
- `消息出速率`
- `复制异常数`
- `Listener 状态`
- `CPU`
- `内存`
- `磁盘`
- `最后心跳`

### 详情抽屉建议

抽屉内部分为 4 个 tab：

- `概况`
- `趋势`
- `事件`
- `关联对象`

各 tab 内容如下：

`概况`

- 节点地址
- 版本
- 启动时长
- 数据目录
- 当前角色负载

`趋势`

- 最近 1h CPU
- 最近 1h 内存
- 最近 1h 连接数
- 最近 1h 消息吞吐

`事件`

- 最近掉线恢复
- leader 承载变化
- listener 异常

`关联对象`

- leader groups 列表
- 异常 groups 列表

## 5.3 Groups

`Groups` 页是按复制组巡检的一致性健康页面。

### 页面结构

- 顶部汇总区
- 中部高密度 group 表
- 右侧详情抽屉

### 顶部汇总指标

- group 总数
- 无 leader 数
- 异常 group 数
- 最大 lag
- leader 分布偏斜度

### 主表格字段建议

- `Group ID`
- `Leader`
- `Peers / 副本数`
- `健康状态`
- `Commit Lag`
- `Fetch / Append 异常`
- `最近 leader 变更时间`
- `所属热点节点`
- `消息写入速率`
- `风险标签`

### 详情抽屉建议

`概况`

- group 成员
- leader / follower 角色
- 当前健康结论

`复制状态`

- 各副本进度
- 最大位点差
- 最慢副本

`稳定性`

- 最近 1h leader 变更次数
- proposal / commit 异常趋势

`流量`

- 该 group 写入趋势
- 关联热点 channel

`关联对象`

- 跳转到 leader 节点
- 查看相关事件列表

## 5.4 Gateway

`Gateway` 页用于确认问题是否发生在接入面，而不是集群复制面。

### 建议查看内容

- listener 列表与状态
- 当前连接数
- 连接建立速率
- 连接断开速率
- 在线会话数
- 认证失败数
- 写队列满异常数
- 会话关闭异常数
- 按节点分布的接入压力

### 页面结构

- 顶部概览卡
- listener 表格
- 节点接入压力表
- 关键短趋势图

## 5.5 Traffic

`Traffic` 页专门观察消息面。

### 建议查看内容

- 发送请求量
- 投递成功率
- ack 速率
- 失败率
- 堆积信号
- 热点节点
- 热点 groups / channels

### 页面结构

- 指标卡
- 4 张短趋势图
- `热点节点` 表
- `热点 groups / channels` 表

## 5.6 Events

`Events` 页是值守闭环页面，用于回答“刚刚发生了什么”。

### 事件类型建议

- 节点掉线
- 节点恢复
- leader 频繁切换
- group lag 超阈值
- listener 异常
- 流量突升或突降

### 页面结构

- 时间线
- 可过滤事件表

过滤维度建议包括：

- 节点
- group
- 严重级别
- 时间窗
- 事件类型

## 5.7 Topology

`Topology` 第一版坚持只读。

### 建议查看内容

- 集群节点配置清单
- group 到 peers 的分布
- gateway listeners 配置
- 生效参数与来源

### 页面结构

- 配置快照卡片
- 节点 / group 分布图
- 参数对照表

## 6. 视觉与交互原则

## 6.1 整体风格

建议采用：

- 冷静
- 专业
- 高密度
- 偏控制台

明确避免：

- 大屏式展示
- SaaS 营销感视觉
- 过度装饰的拓扑图

## 6.2 布局

- 左侧固定导航
- 顶部窄状态栏
- 主体采用 `卡片 + 表格 + 详情抽屉`
- 桌面端优先

## 6.3 色彩语义

基调建议使用中性深灰蓝或石墨色体系，把彩色留给状态语义：

- 绿色：`healthy`
- 黄色：`degraded`
- 红色：`critical`
- 蓝色：`info / selected`

要求：

- 不把高饱和颜色用于纯装饰
- 异常状态必须在全局统一表达

## 6.4 信息密度

- 默认允许高密度表格
- 一屏优先多展示节点与 groups
- 默认列精简，但关键字段不能省
- 支持排序、筛选、快速搜索、保存视图

## 6.5 交互原则

- 首页卡片全部可下钻
- 列表页优先使用右侧详情抽屉，而不是频繁跳页
- 抽屉内部再分 tab，保持上下文连续
- 任何状态尽量同时给出：
  - 当前值
  - 变化趋势
  - 风险标签

## 7. 数据模型与后端接口假设

第一版 UI 不应直接依赖当前零散的内部模块对象，而应面向一层 `Admin Observability API`。

这层接口的职责是聚合以下数据域：

- 节点运行状态
- group / leader / lag 状态
- gateway listener 与连接指标
- 在线会话指标
- 消息吞吐、投递、ack、失败指标
- 事件流与告警流
- 配置与拓扑快照

对应到 UI，建议抽象为以下页面级数据模型：

- `ClusterSummary`
- `NodeSummary`
- `NodeDetail`
- `GroupSummary`
- `GroupDetail`
- `GatewaySummary`
- `TrafficSummary`
- `EventRecord`
- `TopologySnapshot`

UI 设计阶段先以这些观测聚合模型为准，不以当前仓库里某个具体内部结构为准。

## 8. 权限边界

第一版只做只读巡检。

建议先保留两个角色概念：

- `viewer`
- `operator`

其中：

- `viewer` 可以查看所有页面
- `operator` 仅作为未来预留角色

第一版不开放任何危险操作，因此 UI 中不出现以下动作按钮：

- 节点上下线
- 节点摘流
- group 重平衡
- listener 开关
- 会话踢除

## 9. 异常、空状态与采集失败表达

后台值守页面必须严格区分三种状态：

- 值为 `0`
- `无数据`
- `采集失败`

如果不区分，会直接误导巡检判断。

统一要求如下：

- 节点离线要显示原因和持续时间
- group 无 leader 要显示严重级别和最近变更时间
- listener 异常要给出当前状态与最近错误摘要
- 趋势图要明确标记缺口和采集失败区间
- 暂未接入的指标统一显示 `数据暂不可用`，不能用伪造的零值代替

## 10. 第一版明确不做

为了保证范围稳定，第一版明确不做以下内容：

- 节点上下线、摘流、重平衡等写操作
- 用户与会话运营后台
- 深度消息追踪
- 长周期容量规划
- 多环境多集群视角
- 自定义报表系统
- 复杂审批、审计与变更流程

## 11. 设计优先级

第一版实现优先级建议如下：

1. `Dashboard`
2. `Nodes`
3. `Groups`
4. `Events`
5. `Gateway`
6. `Traffic`
7. `Topology`

这个顺序体现了巡检值守的真实使用频率：

- 先发现异常
- 再看节点和复制组
- 再补齐上下文事件
- 最后查看接入流量与配置真相

## 12. 后续 pencil 设计建议

基于本设计文档，下一步用 pencil 出图时建议按以下顺序推进：

1. 先画全局布局框架与导航
2. 再画 `Dashboard`
3. 再画 `Nodes` 与 `Groups`
4. 最后补 `Gateway / Traffic / Events / Topology`

这样可以先把巡检主路径做实，再逐步补全辅助页面。

## 13. 结论

这套后台第一版本质上是一个：

- `单集群`
- `SRE 导向`
- `观测优先`
- `高信息密度`
- `以巡检值守为核心`

的分布式管理工作台。

它不追求第一天就具备完整控制面，而是优先把以下能力做对：

- 快速发现异常
- 快速定位到节点或 group
- 用短趋势判断是否持续恶化
- 用事件流补齐值守上下文

这会是后续 pencil 视觉设计和管理 API 设计的统一基线。
